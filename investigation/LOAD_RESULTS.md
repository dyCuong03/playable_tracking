# LOAD_RESULTS — qa-load-tester

Real, in-process runs of the full pipeline (controller → disk → dispatcher bridge →
fake-redis stream → bigquery worker → mock BQ sink). Generated by
`node scripts/load-pipeline.js`. All counts are observed at the data level (XADD counts,
worker deliveries, mock-sink inserts), not parsed from logs.

Reconciliation invariant: **accepted HTTP == inserted + failed + dedup skipped**, and
**LOST == 0**.

## Failure-mode runs (from `tests/pipeline-integration.spec.js`)

These prove zero loss under adverse conditions (counts asserted in-test, all PASS):

| Scenario | Behaviour observed |
|----------|--------------------|
| Redis outage during accept (40 events) | all 40 accepted (HTTP 200, disk), enqueue threw while down, all 40 drained + inserted after recovery — 0 lost |
| Worker crash before ack (30 events) | 30 left pending in dead consumer's PEL, reclaimed via XAUTOCLAIM by the restarted worker, all 30 inserted — 0 lost |
| Transient BQ failure (8 events) | first insert attempt failed, whole batch retried, all 8 inserted on retry, 0 rejected — 0 lost |
| Partial BQ failure (3 events: ok/retry/reject) | ok+retry inserted (2), invalid row routed to `pixel:rejected` (1) and logged `bigquery_status:"failed"` + `bigquery-insert-failed` — 0 silently lost |

## Notes
- `redis enqueued` counts true `XADD`s to `pixel:events`; the bridge reads every accepted
  item off disk, and dedupe (when applicable) is what would reduce XADDs below items read.
  On unique load `dedup skipped == 0`, so XADD == accepted.
- Run timing (~2s for 5k) is in-process controller-driven (not network HTTP); the goal of
  the harness is correctness/accounting, not throughput benchmarking. HTTP-layer fidelity
  is covered separately by the supertest specs in `tests/pixel.spec.js`.
