# LOAD2.md — Phase 2 load + stuck-state reconciliation

Real in-process runs (controller → disk → dispatcher bridge → fake-redis stream → worker →
mock BQ). Generated by `node scripts/load-pipeline.js` (now with a `disk persisted` column).
Counts are observed at the data level (disk NDJSON items, XADD counts, worker deliveries,
mock-sink inserts), not parsed from logs.

Invariant: **accepted HTTP == disk persisted == redis enqueued == worker consumed ==
inserted + failed + dedup skipped**, **LOST == 0**.

## Stuck-state runs (from `tests/stuck-state.spec.js`, all PASS, zero loss)

| Scenario | Observable symptom | Recovery |
|----------|--------------------|----------|
| Dispatcher stopped (50 events) | HTTP 200; `getQueueStats` shows disk backlog > 0; Redis XADD = 0; nothing inserted | on restart `drainDiskToRedis` bridges all 50, worker inserts 50, disk = 0 — **0 lost** |
| Worker stopped (40 events) | `XLEN pixel:events` = 40 (depth visible); nothing inserted | worker restart drains all 40, stream = 0 — **0 lost** |
| Redis down then recovery (35 events) | HTTP still 200 (disk); enqueue throws (`FAKE_REDIS_DOWN`), nothing silent | after recovery all 35 drain + insert — **0 lost** |

## Notes
- `disk persisted` = items read off disk by the bridge; on unique load it equals
  `redis enqueued` (dedupe would only reduce XADDs below it for true duplicates).
- Timing (~2–5s for 5k) is in-process controller-driven; the harness measures
  correctness/accounting, not throughput.
